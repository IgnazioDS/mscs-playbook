# Timed Automata Basics

## Key Ideas

- A timed automaton extends a finite transition system with real-valued clocks
  that all advance uniformly with time; it models deadlines and durations, but
  not general continuous dynamics such as acceleration or temperature drift.
- Timed automata are strictly more expressive than finite automata while still
  retaining decidable reachability through a finite quotient construction.
- The core expressiveness trade-off is this: clocks give dense-time semantics,
  but only through simple linear constraints and resets. That restriction is
  what keeps reachability PSPACE-complete rather than undecidable.
- The region graph proves decidability in theory; zone-based algorithms using
  Difference Bound Matrices (DBMs) make verification practical in tools such as
  UPPAAL.
- Networks of timed automata provide a practical model of concurrent real-time
  systems, especially protocols, schedulers, and embedded controllers.

## Intuition

A plain finite automaton can say that an acknowledgement eventually follows a
request, but it cannot say whether the acknowledgement arrived within 100
milliseconds. For many autonomous and embedded systems, that difference is the
entire safety question. Airbags, watchdog timers, mutual-exclusion protocols,
and control loops are all sensitive not just to event order but to elapsed time.

Timed automata solve this by adding clocks. A clock is a variable that measures
how much real time has passed since some event. All clocks increase at the same
rate as physical time. A transition may be guarded by a condition such as "take
this edge only when `x >= 2`", and taking that transition may reset some clocks
to zero. A location may also carry an invariant such as "`x <= 5`", meaning the
system is not allowed to remain there once five time units have elapsed.

The important conceptual point is that this yields a finite description of an
infinite-state system. The clock values are real numbers, so there are
infinitely many valuations, yet the structure of the allowed comparisons is
restricted enough that many valuations behave equivalently for reachability.
That is the idea behind the region graph.

## 1. Formal Definition

**Definition (Timed Automaton).**  
A timed automaton is a tuple

`A = (Loc, Loc_0, Act, Clk, Inv, E)`

where:

- `Loc` is a finite set of locations;
- `Loc_0 ⊆ Loc` is the set of initial locations;
- `Act` is a finite set of actions;
- `Clk = {x_1, ..., x_n}` is a finite set of clocks;
- `Inv : Loc -> Φ(Clk)` assigns an invariant to each location;
- `E ⊆ Loc × Act × Φ(Clk) × 2^Clk × Loc` is the set of edges.

Each edge `e = (ℓ, a, g, r, ℓ')` consists of:

- source location `ℓ`;
- action `a`;
- guard `g`, a clock constraint that must hold to fire the edge;
- reset set `r ⊆ Clk`, the clocks reset to zero;
- target location `ℓ'`.

Gloss:

- Locations are the discrete modes.
- Clocks store elapsed time since relevant events.
- Guards enable action transitions only when timing constraints hold.
- Invariants bound how long time may elapse in a location.
- Resets restart timing measurements.

Constraint note:

- The semantics only allows time to elapse while the invariant remains true.
- Guard and invariant satisfiability must be checked together; otherwise a model
  can contain syntactically present but semantically impossible transitions.

Minimal reusable example:

- `Loc = {Idle, Wait, Alarm}`
- `Loc_0 = {Idle}`
- `Act = {request, timeout}`
- `Clk = {x}`
- `Inv(Wait) = (x <= 3)`, all other invariants are `true`
- edges:
  `(Idle, request, true, {x}, Wait)` and
  `(Wait, timeout, x >= 3, ∅, Alarm)`.

## 2. Clock Constraints and Semantics

Clock constraints are generated by:

- atomic constraints `x ~ c` or `x - y ~ c`,
- where `x, y ∈ Clk`, `c ∈ Z`, and `~ ∈ {<, <=, =, >=, >}`.

A clock valuation is a function `ν : Clk -> R_{>=0}`. It satisfies a guard or
invariant in the obvious way, for example:

- `ν ⊨ x <= 3` iff `ν(x) <= 3`;
- `ν ⊨ x - y < 2` iff `ν(x) - ν(y) < 2`.

For `d ∈ R_{>=0}`, define `ν + d` by `(ν + d)(x) = ν(x) + d`. For a reset set
`r ⊆ Clk`, define `ν[r := 0]` by resetting every clock in `r` to zero and
leaving the others unchanged.

The semantic states of a timed automaton are pairs `(ℓ, ν)` with `ℓ ∈ Loc` and
`ν ⊨ Inv(ℓ)`.

There are two kinds of transitions:

- **Delay transition:** `(ℓ, ν) --d--> (ℓ, ν + d)` for `d ∈ R_{>=0}` iff for all
  `0 <= d' <= d`, `ν + d' ⊨ Inv(ℓ)`.
- **Action transition:** `(ℓ, ν) --a--> (ℓ', ν[r := 0])` iff there exists an
  edge `(ℓ, a, g, r, ℓ') ∈ E` such that `ν ⊨ g` and `ν[r := 0] ⊨ Inv(ℓ')`.

Thus a timed automaton induces a timed transition system with infinitely many
states because the valuation component ranges over reals.

## 3. Region Graph and Decidability

Two clock valuations are region-equivalent when they agree on the integer parts
of clocks up to the maximum constant `c_max`, on which clocks have zero
fractional part, and on the ordering of the fractional parts. Alur and Dill
showed that only finitely many such regions exist.

The quotient of the semantic state space by this equivalence is the
**region graph**. Reachability in the timed automaton reduces to reachability
in this finite graph.

The size of the region graph is bounded by

`O(|Loc| * |Act| * (2|Clk|)! * (2*c_max + 2)^|Clk|)`.

That bound is large, which is why explicit region construction is mainly a
proof device rather than the implementation strategy of practical tools.

**Theorem (Alur-Dill reachability).**  
Reachability for timed automata is decidable and PSPACE-complete.

Plain-English reading: despite the infinite real-valued state space, reachability
can still be decided algorithmically, but the worst-case problem is expensive.

*Proof sketch.* The region equivalence partitions the infinite valuation space
into finitely many classes that preserve the truth of guards, invariants, and
future reachability. Construct the finite region graph and run ordinary graph
search on it. PSPACE-hardness follows by reduction from linear-space automata;
PSPACE membership follows because the search can be performed without storing
the full graph explicitly. Full proof in Alur and Dill 1994.

## 4. Zone-Based Algorithms and DBMs

Practical tools such as UPPAAL do not enumerate regions. They represent convex
sets of valuations called **zones**:

- conjunctions of bounds `x - y <= c`;
- stored efficiently as a Difference Bound Matrix.

A DBM supports:

- delay closure;
- guard intersection;
- clock reset;
- emptiness checking;
- canonical normalization.

Zone algorithms are typically much faster in practice because they manipulate
whole symbolic sets of valuations at once instead of single regions.

## 5. Networks of Timed Automata

Real systems are often modeled as parallel compositions of smaller automata.
Each component has local locations and clocks. Components synchronize on shared
action labels, producing a global configuration whose location is a tuple.

Composition is attractive because it matches modular design, but it also
creates state explosion: even if each component is small, the product of their
locations and symbolic clock zones can grow rapidly.

## 6. Worked Example: Timed Mutual Exclusion

We model two processes competing for a critical section and a separate lock
automaton mediating access.

### 6.1 Component automata

For each process `P_i` with `i ∈ {1, 2}`:

- `Loc_i = {Idle_i, Wait_i, Crit_i}`
- `Loc_{0,i} = {Idle_i}`
- `Act_i = {req_i, enter_i, exit_i}`
- `Clk_i = {x_i}`
- `Inv(Wait_i) = (x_i <= 2)`
- edges:
  `(Idle_i, req_i, true, {x_i}, Wait_i)`
  `(Wait_i, enter_i, x_i >= 1, ∅, Crit_i)`
  `(Crit_i, exit_i, true, ∅, Idle_i)`

The lock automaton `L` has no clocks:

- `Loc_L = {Free, Reserved1, Held1, Reserved2, Held2}`
- `Loc_{0,L} = {Free}`
- actions `req_1, enter_1, exit_1, req_2, enter_2, exit_2`
- edges:
  `Free --req_1--> Reserved1 --enter_1--> Held1 --exit_1--> Free`
  `Free --req_2--> Reserved2 --enter_2--> Held2 --exit_2--> Free`

### 6.2 Reachable region graph fragment

Because the lock has only one free token, the reachable global quotient states
fall into a small pattern:

| Region state | Composite location | Clock region | Meaning |
|--------------|--------------------|--------------|---------|
| `q0` | `(Idle_1, Idle_2, Free)` | `x_1 = x_2 = 0` | initial |
| `q1` | `(Wait_1, Idle_2, Reserved1)` | `0 <= x_1 < 1` | process 1 waiting |
| `q2` | `(Crit_1, Idle_2, Held1)` | clocks irrelevant | process 1 in critical section |
| `q3` | `(Idle_1, Wait_2, Reserved2)` | `0 <= x_2 < 1` | process 2 waiting |
| `q4` | `(Idle_1, Crit_2, Held2)` | clocks irrelevant | process 2 in critical section |

There is no reachable region whose composite location contains both `Crit_1`
and `Crit_2`, because the lock automaton has no location that simultaneously
permits `Held1` and `Held2`.

### 6.3 Executable trace

| Step | Location tuple | Clock vals | Type | Guard / invariant check | Action / delay |
|------|----------------|------------|------|-------------------------|----------------|
| 0 | `(Idle_1, Idle_2, Free)` | `x_1 = 0, x_2 = 0` | initial | all invariants true | start |
| 1 | `(Wait_1, Idle_2, Reserved1)` | `x_1 = 0, x_2 = 0` | action | `req_1` synchronizes with lock | `req_1` |
| 2 | `(Wait_1, Idle_2, Reserved1)` | `x_1 = 1.2, x_2 = 1.2` | delay | `x_1 <= 2` holds | `d = 1.2` |
| 3 | `(Crit_1, Idle_2, Held1)` | `x_1 = 1.2, x_2 = 1.2` | action | guard `x_1 >= 1` holds | `enter_1` |
| 4 | `(Idle_1, Idle_2, Free)` | `x_1 = 1.2, x_2 = 1.2` | action | exit enabled | `exit_1` |
| 5 | `(Idle_1, Wait_2, Reserved2)` | `x_1 = 1.2, x_2 = 0` | action | `req_2` synchronizes with lock | `req_2` |

### 6.4 Property check

The mutual-exclusion property is

`A[] not (P1.Crit and P2.Crit)`.

The trace satisfies it, and the reachable region graph fragment explains why:
the lock's location is either `Free`, `Reserved1`, `Held1`, `Reserved2`, or
`Held2`. None of those admits simultaneous ownership by both processes.

## 7. UPPAAL and Property Queries

UPPAAL models timed automata as:

- templates with locations, invariants, and edges;
- channels for synchronization;
- integer variables and clocks;
- TCTL-like queries.

Common query forms:

- `A[] p`: on all reachable states, `p` holds.
- `E<> p`: there exists a path reaching a state where `p` holds.
- `A<> p`: on all paths, `p` is eventually reached.
- `p --> q`: whenever `p` occurs, `q` eventually follows.

Interpretation matters:

- a satisfied `A[]` query is a global invariant over the model;
- a failed query produces a diagnostic trace;
- a query only proves something about the model, not about omitted dynamics.

### Deadline requirements

A requirement such as "within `T` time units of event `A`, event `B` must
occur" is often handled by an observer clock:

- reset clock `x` when `A` occurs;
- move to an error location if `x > T` before `B`;
- verify `A[] not Error`.

## 8. Common Mistakes

1. Modeling general continuous dynamics with timed automata. Clocks always have
   derivative `1`; they are not arbitrary state variables.
2. Forgetting invariants, thereby allowing time to diverge in a waiting state
   that was meant to enforce a deadline.
3. Reading a zone-based diagnostic trace as if it were a unique concrete run;
   symbolic traces may represent many valuations.
4. Building a network whose synchronization labels do not match, silently
   disconnecting components.
5. Ignoring Zeno-like urgent cycles created by zero-time transitions.

## 9. Practical Checklist

- [ ] List every clock and the event whose elapsed time it measures.
- [ ] Check guards and invariants together for satisfiability.
- [ ] Confirm that deadline properties are encoded by observers or queries that
      actually mention time.
- [ ] Use symbolic zones or DBMs in practice rather than explicit region
      construction.
- [ ] Inspect whether composition introduces urgent cycles or deadlocks.
- [ ] Validate the model on at least one accepted and one rejected timed trace.
