# LTL Basics and Trace Semantics

## Key Ideas

- Linear Temporal Logic (LTL) specifies properties of single infinite paths; it
  models ordering and recurrence over time, but it does not directly quantify
  over branching alternatives the way CTL does.
- The formal semantics is an inductive definition of `¤Ç, i ÔŐĘ ¤ć`; informal
  glosses like "always" and "eventually" are only shorthand for that
  definition.
- LTL model checking reduces to automata emptiness: a transition system
  satisfies `¤ć` exactly when the product with a B├╝chi automaton for `┬Č¤ć` has no
  accepting run.
- The complexity is PSPACE-complete in the formula size and linear in the size
  of the transition system; the exponential blow-up comes from the property, not
  from graph search on the model.
- Vacuity and mismatched assumptions are common failure modes: a formula may be
  syntactically correct yet prove almost nothing about the intended system.

## Intuition

Reactive systems do not just need state predicates such as "the door is open"
or "the alarm is off." They need temporal promises: if a request is issued, a
grant eventually follows; once the system enters an emergency mode, it stays
there until reset; a sensor heartbeat occurs infinitely often. LTL was invented
to write those promises over executions of a system.

LTL takes the viewpoint of a single run through time. You imagine one infinite
path and ask whether the formula is true along that path. This is why the logic
is called linear-time: it reasons about one timeline at a time. A model
checker then lifts that path-level notion to a system-level question by asking
whether all paths of the model satisfy the formula.

The power of LTL comes from a small set of temporal operators with precise
semantics. The danger is that they are easy to read too casually. "Eventually"
does not mean "soon," and "always eventually" is very different from
"eventually always." The only safe way to reason with LTL is to return to the
formal satisfaction relation.

## 1. Syntax

LTL formulas are generated by:

```bnf
¤ć ::= true | false | p | ┬Č¤ć | ¤ć Ôłž ¤ł | ¤ć ÔłĘ ¤ł | ¤ć Ôćĺ ¤ł
     | X ¤ć | F ¤ć | G ¤ć | ¤ć U ¤ł | ¤ć W ¤ł | ¤ć R ¤ł
```

where `p` ranges over atomic propositions in `AP`.

**Definition (LTL formula).**  
An LTL formula is any expression generated by the grammar above.

Gloss:

- `X ¤ć`: `¤ć` holds at the next position.
- `F ¤ć`: `¤ć` holds at some future position.
- `G ¤ć`: `¤ć` holds at every future position.
- `¤ć U ¤ł`: `¤ć` holds until `¤ł` holds, and `¤ł` must eventually hold.
- `¤ć W ¤ł`: weak until; `¤ł` need not ever hold.
- `¤ć R ¤ł`: release; `¤ł` must hold until and including the point where `¤ć`
  becomes true, if that ever happens.

## 2. Semantics over Infinite Paths

Let `TS = (S, Act, Ôćĺ, I, AP, L)` be a transition system. Let
`¤Ç = s_0, s_1, s_2, ...` be an infinite path. The satisfaction relation
`¤Ç, i ÔŐĘ ¤ć` is defined inductively.

**Definition (LTL trace semantics).**  
For every infinite path `¤Ç` and position `i >= 0`:

- `¤Ç, i ÔŐĘ true` always;
- `¤Ç, i ÔŐş false` always;
- `¤Ç, i ÔŐĘ p` iff `p Ôłł L(¤Ç[i])`;
- `¤Ç, i ÔŐĘ ┬Č¤ć` iff not `¤Ç, i ÔŐĘ ¤ć`;
- `¤Ç, i ÔŐĘ ¤ć Ôłž ¤ł` iff `¤Ç, i ÔŐĘ ¤ć` and `¤Ç, i ÔŐĘ ¤ł`;
- `¤Ç, i ÔŐĘ ¤ć ÔłĘ ¤ł` iff `¤Ç, i ÔŐĘ ¤ć` or `¤Ç, i ÔŐĘ ¤ł`;
- `¤Ç, i ÔŐĘ ¤ć Ôćĺ ¤ł` iff `¤Ç, i ÔŐş ¤ć` or `¤Ç, i ÔŐĘ ¤ł`;
- `¤Ç, i ÔŐĘ X ¤ć` iff `¤Ç, i + 1 ÔŐĘ ¤ć`;
- `¤Ç, i ÔŐĘ F ¤ć` iff there exists `j >= i` such that `¤Ç, j ÔŐĘ ¤ć`;
- `¤Ç, i ÔŐĘ G ¤ć` iff for all `j >= i`, `¤Ç, j ÔŐĘ ¤ć`;
- `¤Ç, i ÔŐĘ ¤ć U ¤ł` iff there exists `j >= i` such that `¤Ç, j ÔŐĘ ¤ł` and for all
  `k` with `i <= k < j`, `¤Ç, k ÔŐĘ ¤ć`;
- `¤Ç, i ÔŐĘ ¤ć W ¤ł` iff either `¤Ç, i ÔŐĘ ¤ć U ¤ł` or for all `j >= i`, `¤Ç, j ÔŐĘ ¤ć`;
- `¤Ç, i ÔŐĘ ¤ć R ¤ł` iff for all `j >= i`, if for all `k` with `i <= k < j`,
  `¤Ç, k ÔŐş ¤ć`, then `¤Ç, j ÔŐĘ ¤ł`.

We write `¤Ç ÔŐĘ ¤ć` as shorthand for `¤Ç, 0 ÔŐĘ ¤ć`.

A transition system satisfies `¤ć`, written `TS ÔŐĘ ¤ć`, if every infinite path
starting from an initial state satisfies `¤ć`.

## 3. Derived Equivalences

**Proposition (Standard LTL equivalences).**  
For every path `¤Ç` and position `i`:

- `F ¤ć Ôëí true U ¤ć`
- `G ¤ć Ôëí false R ¤ć`
- `¤ć W ¤ł Ôëí (¤ć U ¤ł) ÔłĘ G ¤ć`
- `G F ¤ć Ôëí ┬ČF G ┬Č¤ć`

*Proof sketch.* Each equivalence follows by expanding the semantic definitions.
For example, `¤Ç, i ÔŐĘ true U ¤ć` means there exists `j >= i` such that
`¤Ç, j ÔŐĘ ¤ć` and `true` holds at all earlier positions, which is exactly the
definition of `F ¤ć`. The release equivalence is analogous. The duality
`G F ¤ć Ôëí ┬ČF G ┬Č¤ć` is a De Morgan law over temporal quantification: "infinitely
often `¤ć`" is equivalent to "it is not the case that eventually `┬Č¤ć` holds
forever." Full inductive proofs are standard in Pnueli 1977 and later texts.

## 4. The LTL Model-Checking Problem

The decision problem is:

Given a finite transition system `TS` and an LTL formula `¤ć`, decide whether
`TS ÔŐĘ ¤ć`.

The automata-theoretic solution proceeds in four steps:

1. Translate `┬Č¤ć` into a B├╝chi automaton `A_{┬Č¤ć}`.
2. Compute the synchronous product `TS ÔŐŚ A_{┬Č¤ć}`.
3. Check whether the product has an accepting run, equivalently a reachable
   cycle containing an accepting state.
4. If such a cycle exists, project it to a counterexample path of `TS`; if not,
   then `TS ÔŐĘ ¤ć`.

**Theorem (Automata-theoretic characterization).**  
Let `TS` be a finite transition system and `¤ć` an LTL formula. Then

`TS ÔŐĘ ¤ć` iff the product `TS ÔŐŚ A_{┬Č¤ć}` has no accepting run.

Plain-English reading: the system satisfies the property exactly when no system
execution is accepted by the automaton representing the bad behaviors.

*Proof sketch.* The B├╝chi automaton `A_{┬Č¤ć}` is constructed so that it accepts
exactly the traces satisfying `┬Č¤ć`. The product runs the system and the
property automaton in lockstep, so an accepting run of the product is precisely
a path of `TS` whose label sequence violates `¤ć`. Conversely, any violating path
induces an accepting run in the product. Full proof in standard references such
as Vardi and Wolper's automata-theoretic treatment and Baier and Katoen.

The complexity of LTL model checking is PSPACE-complete in `|¤ć|` and linear in
the size of `TS`.

## 5. Worked Example: `G (req -> F grant)`

Consider the transition system `TS_req` with states:

- `s_0`: idle, `L(s_0) = Ôłů`
- `s_1`: request pending, `L(s_1) = {req}`
- `s_2`: granted, `L(s_2) = {grant}`
- `s_3`: starvation loop, `L(s_3) = {req}`

Transitions:

- `s_0 -> s_1`
- `s_1 -> s_2`
- `s_2 -> s_0`
- `s_1 -> s_3`
- `s_3 -> s_3`

The property is

`¤ć = G (req -> F grant)`.

### 5.1 A B├╝chi automaton for `┬Č¤ć`

Using the equivalence

`┬ČG(req -> F grant) Ôëí F(req Ôłž G ┬Čgrant)`,

we can build a B├╝chi automaton with states:

- `q_0`: no pending violation chosen yet
- `q_1`: a request has been seen; waiting to confirm perpetual lack of grant
- `q_acc`: accepting state meaning "from here on, no grant occurs"
- `q_dead`: grant occurred after the bad guess

Representative transitions:

- from `q_0`, stay in `q_0` on labels without `req`; move to `q_1` on labels
  containing `req`;
- from `q_1`, move to `q_0` if `grant` occurs; stay in `q_1` if not; and
  nondeterministically move to `q_acc` when we guess the current suffix has no
  future grant;
- from `q_acc`, stay in `q_acc` on labels without `grant`; move to `q_dead` if
  `grant` appears;
- `q_dead` is a rejecting sink.

Only `q_acc` is accepting.

### 5.2 Product search

The bad branch of the system is:

`s_0, s_1, s_3, s_3, s_3, ...`

In the product, this yields the reachable accepting cycle:

`(s_0, q_0) -> (s_1, q_1) -> (s_3, q_acc) -> (s_3, q_acc) -> ...`

because `s_3` is labelled with `req` but not `grant`, so the B├╝chi automaton
can remain in its accepting state forever.

### 5.3 Executable trace and property check

| Step | TS state | Labels | B├╝chi state | Reason |
|------|----------|--------|-------------|--------|
| 0 | `s_0` | `Ôłů` | `q_0` | no request yet |
| 1 | `s_1` | `{req}` | `q_1` | request creates an obligation |
| 2 | `s_3` | `{req}` | `q_acc` | guess that grant will never occur |
| 3 | `s_3` | `{req}` | `q_acc` | still no grant |
| 4 | `s_3` | `{req}` | `q_acc` | accepting cycle |

Since the product has an accepting run, `TS_req ÔŐş G (req -> F grant)`.

## 6. Specification Pitfalls

- **Vacuous satisfaction.** If `req` is never true, then
  `G(req -> F grant)` holds vacuously.
- **Unrealizable liveness.** A formula may require progress that the model's
  environment prevents.
- **Finite-trace confusion.** Standard LTL is defined over infinite paths. Tool
  support for finite traces uses separate semantics.

A good practice is to test a specification against a known-good trace and a
known-bad trace before running the full model checker.

## 7. Common Mistakes

1. Explaining `G` or `F` informally without giving the actual `¤Ç, i ÔŐĘ ¤ć`
   semantics.
2. Mixing LTL with CTL notation, such as writing `AG p` in a page that is
   otherwise about LTL.
3. Confusing `F G p` with `G F p`; one means eventual stabilization, the other
   means infinitely often.
4. Interpreting "eventually" as a bounded deadline. Plain LTL has no metric
   time.
5. Accepting a vacuous proof without checking whether the antecedent ever holds.

## 8. Practical Checklist

- [ ] Define the atomic propositions and the path model before writing formulas.
- [ ] State formulas in standard LTL syntax using `X`, `F`, `G`, `U`, `W`, and
      `R`.
- [ ] Test the formula on one intended-satisfying and one intended-violating
      trace.
- [ ] Check for vacuity, especially in implication-heavy specifications.
- [ ] Use the automata-theoretic view to interpret a model checker's
      counterexample as an accepting cycle.
- [ ] Be explicit when working with finite-trace semantics instead of standard
      infinite-trace LTL.
