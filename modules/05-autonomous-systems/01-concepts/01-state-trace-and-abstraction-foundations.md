# State, Trace, and Abstraction Foundations

## Key Ideas

- Every formalism in this module answers the same four questions: what counts
  as the system state, how state evolves, what an observer can see, and what
  properties are checked over the resulting behavior.
- State is not the same thing as observation. A model may contain internal
  variables that matter for future evolution even when they are invisible in
  the specification.
- Executions, traces, and reachable sets are the common semantic objects shared
  by transition systems, timed automata, LTI systems, and hybrid automata.
- Abstraction is only useful when it preserves the property of interest. For
  safety, over-approximation is typically the sound direction.
- The most common foundational modeling error is choosing a state description
  that is too small, so the future depends on hidden history instead of the
  current state.

## Intuition

Before learning a specific formalism, it helps to know what all the formalisms
have in common. Whether the model is a transition system, an LTI controller, a
timed automaton, or a hybrid automaton, the core workflow is the same: define
the system state, define how it changes, define what observations matter, and
then ask whether the observed behavior satisfies a requirement.

This is where many beginners get into trouble. They jump directly to a tool or
notation and start drawing states or writing matrices before deciding what the
state actually needs to remember. If the next behavior of the model depends on
something that is not encoded in the state, then the model is underspecified.
If the model includes many details that never affect the property, then the
state space becomes harder than necessary.

The right mental model is that a formalism is a particular way of packaging the
same semantic ingredients. A transition system packages them as nodes and
edges. An LTI model packages them as vectors and matrices. A timed automaton
packages them as locations plus clocks. A hybrid automaton packages them as
modes plus differential equations. The mathematics changes, but the underlying
questions do not.

## 1. Generic Modeling Vocabulary

**Definition (State-based model).**  
A state-based model is a tuple

`M = (X, U, Step, X_0, Y, Obs)`

where:

- `X` is the set of states;
- `U` is the set of inputs, actions, or disturbances;
- `Step` is the evolution rule, which maps the current state and input to one
  or more successor states, trajectories, or timed evolutions;
- `X_0 ⊆ X` is the set of initial states;
- `Y` is the observation space;
- `Obs : X -> Y` is the observation map.

Gloss:

- `X` records the information needed to predict future behavior.
- `U` captures what the environment or controller may choose.
- `Step` is the formal semantics of one modeling step.
- `X_0` determines where runs may start.
- `Obs` tells us what part of the internal state is exposed to requirements.

Constraint note:

- A good state description should be Markovian: once the current state and
  input are known, past history should not be needed to determine legal future
  behavior.

Minimal reusable example:

- `X = {idle_hi, move_hi, move_lo, charge_lo, charge_hi}`
- `U = {start, drain, dock, fill}`
- `X_0 = {idle_hi}`
- `Y = 2^{idle, moving, charging, low}`
- `Obs(idle_hi) = {idle}`, `Obs(move_lo) = {moving, low}`, and so on.

## 2. Executions, Traces, and Reachable Sets

An **execution** is a concrete behavior generated by the evolution rule. In a
discrete model it is typically a sequence of states:

`x_0, x_1, x_2, ...`

In a continuous model it may be a trajectory `x(t)`. In a hybrid model it is an
alternating sequence of continuous flows and discrete jumps.

A **trace** is the corresponding observation sequence:

`Obs(x_0), Obs(x_1), Obs(x_2), ...`

for discrete models, or the observed signal over time for continuous models.

The **reachable set** is the set of states that occur on some execution from an
initial state. Reachability is the semantic core of many verification tasks:

- invariance asks whether bad states are reachable;
- liveness asks whether progress can be postponed forever;
- controller synthesis asks from which states a winning strategy exists.

## 3. Discrete Time, Dense Time, and Continuous Evolution

The main model families in this module differ in how `Step` behaves:

| Formalism | State form | Evolution style | Observation style |
|-----------|------------|-----------------|-------------------|
| Transition system | finite symbolic state | discrete successor relation | atomic propositions on states |
| Timed automaton | location plus clock valuation | delays plus action jumps | state predicates on configurations |
| LTI system | real-valued vector | ODE or sampled recurrence | output map `y = Cx + Du` |
| Hybrid automaton | mode plus real-valued state | continuous flow plus guarded jumps | predicates on mode-state pairs |

This difference changes the analysis technique, but not the basic semantics:
all four produce executions, traces, and reachable sets.

## 4. Abstraction and Property Preservation

An abstraction map `alpha : X_C -> X_A` compresses a concrete state space into a
smaller abstract one. The abstract model is an **over-approximation** when
every concrete execution maps to an abstract execution, possibly along with
extra abstract behaviors not present concretely.

**Theorem (Trace equivalence preserves trace properties).**  
Let `M_1` and `M_2` be two models over the same observation space. If they
induce exactly the same set of traces, then for every trace property `P`,

`M_1 ⊨ P` iff `M_2 ⊨ P`.

Plain-English reading: if two models generate the same observable behaviors,
then no trace-based specification can distinguish them.

*Proof sketch.* A trace property is just a predicate on traces. If
`Traces(M_1) = Traces(M_2)`, then every trace accepted or rejected by `P` in one
model is accepted or rejected in the other. Therefore the universal statement
"all traces satisfy `P`" has the same truth value in both models. This is the
basic semantic reason that abstraction, quotienting, and equivalence checking
matter in verification.

The theorem is intentionally simple, but it gives the right engineering rule:
if you abstract away information, make sure the abstraction preserves the
observable behaviors relevant to the property you plan to check.

## 5. Worked Example: Rover Battery Controller

Consider a simplified rover controller with full state set:

- `idle_hi`: rover waiting with healthy battery
- `move_hi`: rover moving with healthy battery
- `move_lo`: rover moving with low battery
- `charge_lo`: rover docked and charging from low battery
- `charge_hi`: rover docked and charging with restored battery

Actions:

- `start`: `idle_hi -> move_hi`
- `drain`: `move_hi -> move_lo`
- `dock`: `move_lo -> charge_lo`
- `fill`: `charge_lo -> charge_hi`
- `stop`: `charge_hi -> idle_hi`

Observation map:

- `Obs(idle_hi) = {idle}`
- `Obs(move_hi) = {moving}`
- `Obs(move_lo) = {moving, low}`
- `Obs(charge_lo) = {charging, low}`
- `Obs(charge_hi) = {charging}`

Initial state:

- `X_0 = {idle_hi}`.

### 5.1 Executable trace

| Step | Full state | Observation | Action taken | Next state |
|------|------------|-------------|--------------|------------|
| 0 | `idle_hi` | `{idle}` | `start` | `move_hi` |
| 1 | `move_hi` | `{moving}` | `drain` | `move_lo` |
| 2 | `move_lo` | `{moving, low}` | `dock` | `charge_lo` |
| 3 | `charge_lo` | `{charging, low}` | `fill` | `charge_hi` |
| 4 | `charge_hi` | `{charging}` | `stop` | `idle_hi` |

### 5.2 Property check

The safety property is:

`G not (moving and charging)`.

The trace satisfies it because no observation contains both `moving` and
`charging`. The example also shows why state and observation are different:

- `move_hi` and `move_lo` are different internal states because future actions
  differ;
- both are still observations of motion from the perspective of a high-level
  safety property.

### 5.3 Abstracted model

If the property only mentions `moving`, `charging`, and `low`, then the exact
battery charge value is irrelevant. We can abstract the state space to:

- `idle`
- `move_ok`
- `move_low`
- `charge_low`
- `charge_ok`

This abstraction preserves the worked-example safety property because it
preserves the same observation traces.

## 7. Common Mistakes

1. Treating visible outputs as if they were the whole state, even when hidden
   variables affect future behavior.
2. Choosing a state that depends on unrecorded history, which makes the model
   non-Markovian.
3. Abstracting away timing or continuous variables before checking whether the
   property depends on them.
4. Confusing an execution with a trace; many different executions may induce the
   same observation trace.
5. Claiming an abstraction is sound without stating what property class it
   preserves.

## 8. Practical Checklist

- [ ] Write down the state variables before choosing a formalism.
- [ ] Ask whether the chosen state is sufficient to predict all legal futures
      given the current input.
- [ ] Separate internal state from observable labels or outputs.
- [ ] Define what one step of evolution means: successor, delay, flow, or jump.
- [ ] Compute at least one concrete execution and its induced observation trace.
- [ ] State explicitly which properties the abstraction is meant to preserve.
